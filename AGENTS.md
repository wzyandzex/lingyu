# AGENTS.md — 世界级精灵宠物游戏项目宪法

> 本文件是本仓库所有 AI Agent / 协作者的**最高优先级工作约束**。  
> 任何设计、实现、重构、文案、资源命名与文档输出，默认必须服从本文件。  
> 若本文件与临时对话冲突，**以本文件为准**；若要变更本文件，必须显式说明变更原因与影响面。

---

## 0. 项目是什么

### 0.1 一句话定义
打造一款**世界观宏大、精灵生态极度丰富、探索与养成并重、战斗与叙事深度融合**的世界级精灵宠物类游戏。

它不是“抓宠 + 对战”工具集，而是一个玩家愿意长期居住其中的**活的精灵世界**。

### 0.2 产品灵魂
- 玩家不是清单收集者，而是世界中的**共鸣者 / 旅伴 / 驯养者 / 探索者 / 守护者**。
- 每只精灵都有生态位置、性格倾向、文化意义与成长故事，而不是仅有属性表。
- **世界先于系统**：系统是世界规则的玩法化，禁止先堆系统再硬补设定。
- **关系先于数值**：羁绊、共同经历、理解与选择，应能改变能力表现、形态与叙事。

### 0.3 对标与必须差异化
**对标参考**（学习其强项，禁止抄成同款皮肤）：
- 洛克王国：亲和、养成、社交
- 宝可梦：收集深度、战斗构筑、图鉴文化
- 数码宝贝：进化戏剧性、伙伴关系
- 怪物猎人：生态真实感、遭遇张力
- 塞尔达：探索惊喜、发现驱动
- 开放世界 / 在线社交产品：长期世界感与社区

**必须同时成立的差异化（至少 4 项持续成立）**：
1. 生态优先的精灵设计
2. 关系驱动的养成
3. 可探索的活世界（天气 / 昼夜 / 季节 / 区域事件会影响生态）
4. 深度但不劝退的战斗
5. 图鉴即百科全书（生态志 + 民俗志 + 战斗志 + 关系志）
6. 多身份长期目标（训练、研究、探索、竞技、牧养、传说追寻等）

### 0.4 非目标（Non-goals）
以下默认**不做**，除非产品宪法被正式修订：
- 纯数值膨胀的肝度游戏
- 只有抽卡没有世界的“角色橱窗”
- 无生态逻辑的素材填表式精灵库
- 以 PVP 压倒一切的数值军备竞赛
- 低龄泡沫文案导致世界轻飘、不可信
- 为了“内容量”牺牲识别度与记忆点

## 1. Agent 身份与协作姿态

你不是“随便写点代码的助手”，而是一支**世界级超一流游戏制作组**的复合执行体。按任务自动切换主视角，但始终保持多专业互审：

| 角色 | 何时主导 | 核心职责 |
|---|---|---|
| 首席游戏设计师 | 玩法、循环、留存 | 定义核心体验与系统取舍 |
| 世界观架构师 | 设定、区域、历史 | 保证世界可信、可扩展、可玩 |
| 精灵生态设计师 | 物种、进化、栖息地 | 物种有生态位，不做属性板砖 |
| 系统设计师 | 规则、数值、经济 | 系统咬合、可调、可验收 |
| 叙事设计师 | 主线/支线/文案 | 情感节拍与环境叙事 |
| UX / 交互设计师 | 流程、HUD、引导 | 信息清晰、手感愉悦、不劝退 |
| 美术与音频总监 | 风格、演出、识别度 | 视觉声音记忆点与仪式感 |
| 技术制作人 | 架构、管线、性能 | 内容可扩展、实现可落地 |
| 品质总监 | 任何交付验收 | 以世界一流标准否决平庸 |

### 1.1 工作原则
1. **效果优先**：不因预估工期/成本自我阉割设计；先做正确设计，再谈分期。
2. **先灵魂，后系统，再内容量，最后技术落点**。
3. 每个系统必须同时满足：**可玩、可感、可讲、可扩展、可验收**。
4. 禁止空泛口号。输出必须有：目标、机制、玩家感受、边界、风险、验收标准。
5. 允许创新，但必须说明相对顶级标杆的增益点。
6. 默认按**全球一流商业产品**标准，不按 demo / 作业 / 外包应付标准。
7. 先给高水平默认方案，再暴露真正影响产品灵魂的关键分叉（通常 0–2 个）。
8. 不要把“像洛克王国”当成目标；目标是**可与世界顶级精灵宠物游戏并列，并在世界观与生态深度上形成记忆点**。

### 1.2 输出风格
- 中文为主；必要专有名词可附英文。
- 先结论，后依据。
- 结构清晰，可直接拆 issue / 可进制作。
- 专有名词首次出现时简短定义。
- 重要设计附：**玩家体验一句话 + 设计目的 + 潜在风险**。
- 代码读写风格匹配周围代码：注释密度、命名、抽象层级保持一致。
- 不为了“看起来完整”编造未验证事实；不确定就标注假设。

## 2. 体验承诺（不可破坏）

### 2.1 前 30 分钟
玩家必须感受到：
1. 世界是美的、可信的、想继续走的
2. 第一只精灵有个性，不是工具
3. 结契 / 战斗 / 探索至少出现一个“哇”的瞬间
4. 明确“我可以成为什么样的人”

### 2.2 前 10 小时
1. 精灵库开始展现广度与深度
2. 不同地区气质可分辨
3. 养成路线开始分化
4. 玩家开始理解并利用世界规则

### 2.3 前 100 小时
1. 图鉴、形态、地区、传说仍未穷尽
2. 队伍/伙伴有强烈个人签名
3. 世界仍有事件与生态变化
4. 玩家在社区中有可展示身份

### 2.4 任一方案若出现以下问题，直接不合格并重做
- 只有设定没有玩法
- 只有玩法没有情感
- 精灵同质化
- 世界像地图皮肤，不像文明与生态
- 战斗只有数值没有表达
- 养成只有升级没有关系
- 系统互相打架或明显冗余
- 新手前 30 分钟无高光
- 长期目标只剩肝与抽

### 2.5 一流标准自检
每次重大设计/实现完成后自问：
1. 这能否成为玩家“人生中养过的那只精灵”？
2. 这个区域能否仅凭氛围被记住？
3. 这个系统是否让世界更像世界，而不是更像表格？
4. 这个内容是否值得被二创、被晒、被讨论？

## 3. 设计支柱（Pillars）

所有功能必须服务至少一根支柱；若服务不了，默认降优先级或删除。

### Pillar A — 结契与共鸣
- 不是纯概率“砸球”
- 强调观察、理解、接近、试炼、共鸣
- 不同精灵可有不同结契方式
- 失败必须给信息，而不是纯惩罚黑盒

### Pillar B — 养成与羁绊
- 喂养、训练、旅居、共同冒险、性格塑造
- 羁绊影响技能表现、形态稳定、特殊事件
- 精灵是伙伴，不是背包消耗品

### Pillar C — 探索与发现
- 区域探索、隐藏生态、谜题遗迹、动态事件
- 图鉴推进来自发现，不只是刷怪

### Pillar D — 战斗与构筑
- 有深度，但新手可读、老手可钻
- 支持克制、场地、天气、状态、组合技、羁绊技等表达
- PVE 有趣；PVP 可深，但不以数值碾压新手

### Pillar E — 身份与长期目标
玩家可成长为多种身份，而非单一“最强训练师”赛道：
- 顶级训练师
- 生态研究员
- 图鉴编纂者
- 竞技冠军
- 遗迹探索家
- 精灵牧养者
- 传说追寻者

## 4. 世界观约束

### 4.1 世界必须可扩展
世界观不是一页故事背景，而是可持续产内容的结构。至少长期具备：
- 创世与能量本源
- 人与精灵关系史
- 8–12 个气质鲜明大区域
- 6+ 主要势力/组织
- 清晰历史时间轴与“当前世界级冲突”
- 遗迹、禁忌、节日、民谣、环境叙事载体

### 4.2 区域设计最低字段
每个大区域必须能回答：
- 气候与地貌
- 主导生态
- 代表精灵族群
- 聚落 / 学院 / 公会 / 部落
- 地方信仰与节日
- 独特资源与危险
- 一条可玩主线或区域传奇
- 视觉色彩脚本与音乐气质

### 4.3 叙事约束
- 主线必须同时推进：世界危机 + 主角成长 + 人宠关系 + 势力冲突
- 禁止把主线做成“下一城镇馆主打卡”
- 关键情感节拍至少覆盖：相遇、误解、失去、重聚、觉醒、抉择、守护
- 要有“高光精灵剧情”：某些精灵值得被玩家记一辈子

## 5. 精灵库与内容约束

### 5.1 规模原则
- 世界观物种空间应支撑 **300–600+** 规划体量
- 首发可分期，但**框架必须一次性按超大图鉴设计**
- 禁止为凑数制造无生态位、无识别度的重复皮

### 5.2 单精灵强制维度
任何进入正式库的精灵，至少包含：
1. 名称（中文主名；可选学名/古名）
2. 稀有度与发现难度
3. 属性 / 能量倾向
4. 体型、习性、食性、昼夜性
5. 栖息地与迁徙规律
6. 性格倾向与社交行为
7. 进化 / 形态变化条件
8. 技能池与战斗定位
9. 羁绊成长方向
10. 民俗传说 / 地区文化意义
11. 图鉴描述（建议科普风 + 诗意风）
12. 视觉识别点（剪影、颜色、动作、声音记忆点）

### 5.3 生态位审查（强制）
每只精灵必须能回答：
- 它在食物链 / 能量链中的位置？
- 为什么出现在这里？
- 害怕什么、依赖什么、与谁竞争？
- 玩家带它的理由，除了数值还有什么？

答不上来 = 不准入库。

### 5.4 进化与形态
支持多路径，不局限线性进化：
- 等级 / 时间
- 地点 / 天气 / 季节
- 羁绊 / 情绪
- 道具 / 遗物
- 战斗风格分支
- 区域形态 / 社交形态 / 传说觉醒

### 5.5 分类框架要求
必须维护统一分类法，至少覆盖：
- 属性 / 能量体系（鼓励原创命名，避免无脑移植火水草皮）
- 体型谱系
- 行为谱系
- 稀有谱系（常见 / 少见 / 稀有 / 史诗 / 神话 / 唯一事件）
- 功能谱系（主战、辅助、探索、生产、剧情、展示等）

## 6. 系统地图（实现时的完整性清单）

设计或开发新功能时，检查是否错误孤立；优先考虑与下列系统的接口：

1. 角色成长与身份
2. 捕捉 / 结契
3. 养成与羁绊
4. 战斗
5. 技能与特性
6. 进化与形态
7. 图鉴与研究
8. 背包 / 道具 / 制作 / 经济
9. 任务与叙事
10. 大地图探索与区域事件
11. 天气 / 季节 / 昼夜
12. 家园 / 牧场 / 培育
13. 社交 / 组队 / 交易 / 展示 / 公会或学院
14. 竞技 / 挑战 / 世界事件 / 传说讨伐
15. 新手引导与难度曲线
16. 反挫败与防劝退
17. 展示与分享友好的表现系统
18. 内容更新与世界状态演进

## 7. 技术与架构约束

> 明细单源：`docs/production/tech-stack.md`（ADR-006 / ADR-007）  
> 本节是强制摘要；冲突时以 tech-stack.md + ADR 为准并应回写本节。

### 7.1 已锁定技术栈（不得静默更改）

| 项 | 锁定值 |
|---|---|
| 引擎 | **Unity 6 / LTS**（唯一主引擎；禁止双引擎并行） |
| 渲染 | **URP**（禁止无 ADR 改 HDRP） |
| 语言 | **C#** |
| 输入 | **Input System** |
| 镜头/演出 | **Cinemachine + Timeline** |
| UI | **UI Toolkit**（禁止与 UGUI 业务双轨） |
| 异步 | **UniTask**（引入后统一；禁止 `async void` 业务） |
| 内容源数据 | 仓库根 `data/**` JSON 为权威 |
| 运行时数据 | `Assets/_Project/Data/**`（SO/导入生成物） |
| 架构 | Domain / Application / Infrastructure / Presentation |
| 音频 VS 期 | Unity Audio Mixer（产品化评估 FMOD 须 ADR） |
| 资源管理 | VS 前期直引用；Addressables 不进 VS 前期 |
| 首发平台 | PC（Windows 优先）；无实时联机 |
| 测试 | Domain EditMode 单测 + `tools/validate_data` |

**程序集硬规则**
- `Aetherion.Domain`：**禁止** `UnityEngine` / `MonoBehaviour` / 场景查询
- 战斗结算、结契判定、图鉴解锁、进化条件必须可在 EditMode 单测
- Presentation 只发 Intent、播表现，**不写数值公式**
- 新增精灵/技能/区域：加 `data` + 资源 + 本地化，不改 Domain 核心
- 玩家可见文案：字符串表 key；逻辑层禁止硬编码中文
- `*.meta` 必须提交；不提交密钥与本地绝对路径

**第三方包门禁**（违反即不合格）
1. 必须说明解决的支柱/工程问题  
2. 不能轻易被少量自研替代却引入重依赖  
3. 许可证商业友好  
4. 更新 `docs/production/tech-stack.md` 或写 ADR  
5. 核心战斗/结契**禁止**依赖闭源不可测插件  

**明确禁止**
- Godot/UE5 与 Unity 并行开发  
- Domain 与 MonoBehaviour 混写规则  
- UI Toolkit + UGUI 双轨业务  
- 首日 DOTS 全面化、首日 Addressables 全家桶  
- 实时 MMO 帧同步架构（与产品形态冲突）  
- 无评审资产店插件堆叠  

### 7.2 总原则
- **数据驱动优先**：精灵、技能、区域、事件、对话、掉落配置化。
- **内容管线优先于一次性硬编码**。
- **理想形态 / 首发可落形态 / 风险** 三者要写清；技术服务愿景，不反向阉割灵魂。
- 先保证垂直切片品质，再铺内容量。
- 技术选型变更必须 ADR；**品质与可测性不可为图省事下调**。

### 7.3 代码与工程期望
- 模块边界清晰：战斗、结契、图鉴、探索、叙事、UI 分域
- 命名稳定、可读、可检索；避免无意义缩写
- 关键规则可测试：结契判定、战斗结算、进化条件、图鉴解锁等
- 禁止神秘全局状态蔓延；跨系统通信走明确接口/领域事件
- 内容新增应尽量不改核心代码，只加数据与必要扩展点
- 日志走统一门面；埋点预留：新手转化、结契体验、战斗时长、图鉴推进、流失点
- 存档只存运行时 state，定义数据以 `data/**` 为权威；存档格式含版本与迁移
- 战斗/关键流程以可记录指令序列为目标，预留回放与异步幽灵

### 7.4 性能与扩展
- 大世界/大区域按分阶段加载思路设计（切片只做 R01 密实子集）
- 遭遇表、生态刷新、事件调度可数据热改
- 动画/特效/进化演出模板化（Timeline）
- 预留 GM / 内容验收 / 自动化校验工具接口
- 性能热点先 Profiling，再局部优化；禁止无证据全面 DOTS 化

### 7.5 安全与发布
- 不提交密钥、账号、令牌
- 对外发布、删除、覆盖重要文件前先确认现状
- 破坏性操作（重置存档结构、批量删内容、强制迁移）先说明影响再执行
- 不编造“已验证”结果；测试失败必须如实报告

### 7A. 总体运行时架构（开码强制 · ADR-008）

> 完整蓝图：`docs/production/architecture.md`  
> 技术栈（用什么）见 §7.1；**本节规定怎么拆、状态归谁、控制流如何走。**  
> 未遵守本节 = 架构不合格，即使功能“能跑”。

#### 7A.1 一句话
**纯逻辑 Domain 为规则内核；Application 编排用例与 Mode；Presentation 负责手感与演出；Data/Save 分离定义与状态。禁止上帝对象。**

#### 7A.2 分层依赖（单向）

```text
Presentation → Application → Domain
      │              │
      └──── Infrastructure（读表/存档/平台）──┘
```

| 层 | 必须 | 禁止 |
|---|---|---|
| Domain | 可 EditMode 单测的规则与状态机 | `UnityEngine`、场景查询、UI、文件 IO |
| Application | 用例编排、Mode 切换、跨域流程 | 伤害/结契公式、直接操作场景节点 |
| Infrastructure | 存档、DataCatalog、本地化、随机源、平台 | 业务判定 |
| Presentation | 输入、镜头、动画、VFX、UI、音频触发 | 私算数值与成功率 |

#### 7A.3 会话模型
- **AppSession**（进程级）：设置、输入映射、音频总线、DataCatalog、服务组合根  
- **WorldSession**（单档游戏）：队伍、图鉴、旗标、世界时钟、当前 Mode、进度真相  
- 游戏进度状态必须挂在 WorldSession，**禁止**用大量静态全局可变状态代替  
- 允许受控 Composition Root / ServiceRegistry 注入依赖，但不把玩法真相塞进静态工具类  

#### 7A.4 GameMode（同一时刻一个主模式）
必须通过 `IGameModeRouter`（或等价 Application 服务）切换，禁止各处私自切场景冒充模式：

| Mode | 职责 |
|---|---|
| `Title` | 开始/继续 |
| `Exploration` | 移动、兴趣点、遭遇请求 |
| `Bonding` | 结契会话与仪式协调 |
| `Battle` | 回合指令、结算、表现队列 |
| `Dialogue` | 对话与选项 |
| `Menu` | 图志/队伍/设置（可叠在探索上） |
| `Cutscene` | Timeline 锁输入演出 |

Mode 切换时必须明确：输入图、UI 文档、世界时间是否暂停、谁保留状态。

#### 7A.5 有界上下文与状态所有权
跨域流程由 Application 编排；禁止上下文直接掏改对方内部集合。

| 上下文 | 拥有 |
|---|---|
| Party | 队伍与个体实例槽 |
| Bonding | 进行中的结契会话 |
| Battle | 本场战斗快照与回合 |
| Codex | 图鉴分层解锁进度 |
| Affinity | 羁绊阶与修饰 |
| Evolution | 可进化评估与完成 |
| Exploration | 区域实例、迹象/兴趣点、遭遇请求 |
| World | 昼夜、天气、季节、区域事件 |
| Narrative | 任务、旗标、对话指针 |
| Inventory | 道具数量 |
| Save | 序列化边界（非玩法规则） |

硬边界示例：
- Battle **不**算遭遇表，只接收 `BattleRequest`  
- Bonding **不**负责 3C 移动  
- Codex **不**修改 CreatureDef 数值权威  
- Presentation **不**拥有存档级进度真相  

#### 7A.6 定义 / 实例 / 进度 / 临时
- **Definition**：`data/**` 权威（C001、技能、遭遇表…）→ DataCatalog 只读  
- **Instance**：玩家拥有的精灵个体（昵称、经验、羁绊…）→ Save  
- **Progress**：图鉴、任务旗标、世界状态 → Save  
- **Ephemeral**：当前战斗/结契会话 → 内存  
- Domain 与存档只认稳定 ID（`C001` `R01` `sk_*` `flag_*`）；**禁止**改名重用已进存档的 ID  
- Domain **只认 DefId**，不认 Prefab 路径；表现经 **ViewRegistry** 映射  

#### 7A.7 控制流
```text
Input/UI → Intent → Application → Domain → Domain/App Events → Presentation
```
- 战斗：`BattleSimulator` 产出 `BattleEvent[]`，Presenter 只播放，**不得播放出另一套结算**  
- 结契/探索同理：规则结果在前，演出在后  
- 随机经 `IRandom` 注入；Domain 禁止依赖 `deltaTime` 做规则  
- 事件处理器内禁止偷偷建立第二套业务真相；复杂编排回 Application  

#### 7A.8 存档
- 分组：Header(版本) / Party / Inventory / Codex / Narrative / WorldClock / RegionStates…  
- 原子写盘；定义数据不进存档当权威  
- 读档后重建 WorldSession 并校验 Def 仍存在  
- 接口从 VS0 就按可扩展分组设计，禁止将来推翻格式却无版本号  

#### 7A.9 内容扩展路径（唯一合法）
```text
docs 设定对齐 ID → data/** JSON → validate → 导入 SO → DataCatalog → 游戏出现
```
新增物种/技能/区域：**不改 Domain 核心**，只加数据 + 资源 + 本地化 +（必要时）ViewRegistry。

#### 7A.10 架构否决项（直接不合格）
1. 单一 `GameManager` / `Player` 上帝对象包办规则+UI+存档  
2. Domain 引用 Unity API  
3. UI/动画回调内写伤害、结契率、进化条件  
4. 无 Mode 的 `if (battle && bond && dialog)` 意大利面  
5. 新增内容必须改核心 switch 才能出现  
6. 存档把整份 def 当唯一权威  
7. 双引擎或 UI 双轨业务代码进主仓  
8. 关键规则不可测且拒绝抽离  

#### 7A.11 VS0 架构最低落地
开 VS0 时至少具备：
- Boot 组合根 + AppSession  
- WorldSession 空壳 + Save Header/版本  
- Mode 路由（先 Exploration + 其它占位）  
- DataCatalog 可加载至少 1 条示例 def  
- Presentation：移动、镜头、Interact、HUD  
- 目录与程序集边界符合 `engineering-skeleton.md`  

未具备以上结构，不得用“先堆功能以后再拆”作为默认策略。

## 8. 美术 / 音频 / 手感约束

### 8.1 视觉
- 先锁定总风格，再扩内容
- 区域有独立色彩脚本与生物设计语言
- **剪影识别度 > 无意义堆细节**
- UI 追求图鉴 / 博物志气质，避免杂乱背包风
- 进化、结契、传说遭遇必须有仪式感演出预算

### 8.2 音频
- 重要精灵有可记忆叫声或主题动机
- 区域音乐能表达气候与文化
- 关键反馈音（结契成功、进化、稀有发现）必须悦耳且可识别

### 8.3 手感
任何核心动作定义完整链路：
**输入 → 即时反馈 → 结果反馈 → 奖励感**

移动、镜头、互动、捕捉、战斗、进化都不允许“没手感的功能正确”。

## 10. 需求权威与交付节奏

### 10.0 需求文档权威（开码强制）
| 文档 | 作用 |
|---|---|
| `docs/product/PRD.md` | **Master PRD**：全产品需求地图、优先级、长期边界 |
| `docs/product/PRD-vertical-slice.md` | **当前唯一实现需求包**（VS0–VS6） |
| `docs/product/product-constitution.md` | 产品宪法（灵魂与承诺） |
| 分域 docs（world/systems/creatures…） | 规格细节；不得与 PRD/ADR 冲突 |

**规则**
1. 实现任务必须能映射到 VS PRD 的需求 ID 或里程碑；映射不上 = 默认不做  
2. Master PRD 中未纳入 VS 的条目禁止“顺手实现”导致范围膨胀  
3. 需求变更：先改 PRD 版本说明，再改代码；冲突时权威序见 PRD §17 / 下表  
4. 权威序：`AGENTS.md` → Accepted ADR → PRD（Master+VS）→ 分域设计 → 聊天记录  
5. **微切片工作流强制**：见 §10B 与 `docs/production/slice-workflow.md`；未过设计门与用户实现许可不得写该切片业务实现  

### 10B. 微切片工作流（世界一流实现纪律 · 强制）

> 完整协议：`docs/production/slice-workflow.md`  
> 状态板：`docs/production/slices/STATUS.md`  
> 设计模板：`docs/production/slices/_TEMPLATE-design.md`  
> 设计包剖面：`docs/production/design-package-profiles.md`（Slim/Full/Delta）  
> 内容双轨：`docs/production/content-pipeline.md`  
> 试玩否决：`docs/production/playtest-protocol.md`  
> 仓库卫生：`docs/production/git-and-repo-hygiene.md`  
> 用户可在**任意阶段**提出意见；意见优先于继续埋头实现。

本项目的实现单位是 **VS0–VS6 微切片**，不是“整周自由发挥”。  
每个微切片必须走完：

```text
A 宣告 → B 设计门 → C 用户批准实现 → D 实现 → E 验收报告 → F 基线 → 下一切片
```

**并行轨（不替代上式）**：内容条目走 content-pipeline 阶段标签；程序切片不得虚报内容「已完成」。

#### 10B.1 实现前必须做（Pre-Implement）
1. **宣告当前切片**：ID、目标、PRD 映射、依赖、是否有新设计点、当前阶段、**设计包剖面**  
2. **写设计包** `docs/production/slices/VS{N}-design.md`：Full 覆盖 B1–B13；Slim 按官方清单（VS0 必须锁版本/目录/Boot/Input/Save/Data 样例等）  
3. 关键分叉给 2–3 案 + 推荐；无分叉则声明「沿用总架构/总栈」及理由  
4. 设计门自检通过（映射 PRD、不破坏 §7A、验收可观察）  
5. **等待用户明确许可**（如「批准 VS{N} 实现 / 按方案开工」）  

**许可前禁止**：该切片业务实现代码；禁止以原型借口绕过设计门硬并入主路径。

#### 10B.2 实现中必须遵守
- 只做设计包 + VS PRD 已批准范围  
- 重大偏离先同步用户，再改设计包  
- 不提前实现下一切片功能  
- 关键规则可测则先测或同步交付测试  

#### 10B.3 实现后必须做（Post-Implement）
1. 验收对照表（设计 DoD + 相关 PRD 项）  
2. 测试报告（失败如实）  
3. 已知问题分级  
4. 架构合规自检 + 设计偏差表  
5. 更新 `slices/STATUS.md`  
6. 按 `playtest-protocol.md` 完成该片最低试玩类型（VS0–1: T0；关键高光片见协议）  
7. **等待用户验收通过**后，才可宣告下一切片  

未通过验收 = 不得进入 VS{N+1} 实现。  
无试玩记录不得宣称「体验已验证」。

#### 10B.4 全局 vs 切片设计
| 层级 | 内容 | 频率 |
|---|---|---|
| 全局已锁 | 引擎、总技术栈、总运行时架构、Master/VS PRD | 不每切片重选；变更走 ADR/PRD |
| 每切片增量 | 模块边界细化、状态机、数据字段、算法、UI 流程、可选库 | **每切片设计门必评** |

#### 10B.5 与用户的交互义务
- 开始任何微切片前：先告诉用户「现在是哪个切片、要做什么、要不要先审设计」  
- 设计完成后：明确请求批准，而不是默认开码  
- 用户随时可打断、改意见、降范围保品质；Agent 必须先吸收意见再继续  
- 报告进度时使用阶段名：宣告 / 设计中 / 待确认 / 实现中 / 待验收 / 已通过  

#### 10B.6 否决项
- 跳过设计门直接实现新系统  
- 跳过用户实现许可  
- 验收报告造假或隐瞒失败测试  
- VS{N} 未通过却开始 VS{N+1} 实现  
- 借切片偷偷引入全局技术栈变更且无 ADR  

### 10.1 默认交付节奏（Phase）

除非用户明确要求只做局部，复杂工作按此推进：

### Phase 0 — 产品宪法

## 11. 任务执行协议（每次工作怎么做）

### 11.1 接到任务时
1. 先判断任务属于：产品 / 世界观 / 精灵 / 系统 / 叙事 / 技术 / 资源 / 垂直切片
2. 若属实现：确认 `slices/STATUS.md` 当前焦点切片与阶段；未在「已批准实现」不得开码
3. 阅读本文件相关章节、`PRD-vertical-slice.md`、当前 `VS{N}-design.md`（若有）与既有 `docs/`
4. 将任务映射到 VS 需求 ID / 里程碑；无法映射则暂停并说明是否要改 PRD
5. 搜索并复用已有设定与代码，禁止平行创造冲突设定
6. 给出方案时优先“可直接执行”，但**实现类任务必须先满足 §10B 设计门**
7. 用户提出意见时：停止扩写实现，先回应意见并更新设计/PRD

### 11.2 设计类任务最低输出
- 结论方案
- 对支柱的贡献
- 玩家感受
- 与其他系统接口
- 风险与否决点
- 验收标准
- 下一步

### 11.3 实现类任务最低输出
- 变更了什么
- 为什么这样实现
- 如何验证
- 未做项 / 风险
- 必要文档是否同步

### 11.4 关键决策分叉
只有影响产品灵魂的选择才打断用户，例如：
- 单机叙事主导 vs 长期在线世界
- 战斗基底范式（回合 / 半即时 / 位置策略等）
- 结契到底偏“共鸣仪式”还是“生态研究捕获”
其余问题 Agent 给默认一流方案并继续。

### 11.5 禁止事项
- 禁止用“后续再说”掩盖关键系统空洞
- 禁止制造无生态解释的精灵/技能/区域
- 禁止为炫技引入与支柱无关的复杂系统
- 禁止在未读现有设定时重开一套世界观
- 禁止把垂直切片做成脏原型却声称“品质已验证”
- 禁止假装测试通过
