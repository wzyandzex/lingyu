# VS 设计包精简规则（Design Package Slim Profile）v1.0

> 状态：强制补充 `slice-workflow.md`  
> 目的：在**不降低可验收性**的前提下，避免 VS0/薄切片被 B1–B13 全量表格拖死  
> 原则：世界一流 = 边界清晰可验收，≠ 每片写小说

---

## 1. 何时用 Full / Slim

| 剖面 | 适用 | 说明 |
|---|---|---|
| **Full** | VS2 结契、VS3 战斗、VS4 进化、含新算法或新 Mode 业务 | B1–B13 全写；算法/状态机必可测 |
| **Slim** | VS0 世界壳、纯基础设施、明确「无新玩法规则」 | 必填子集 + 其余正式 **N/A** |
| **Delta** | VS5/VS6 以内容与打磨为主 | 以变更列表+验收为主，架构无增量则声明沿用 |

选用剖面必须在设计包首页声明。Agent 不得用 Slim 逃避 VS2/VS3 的状态机与公式。

---

## 2. Slim 必填 / 可 N/A

| 章节 | Slim | 备注 |
|---|---|---|
| B1 范围 | **必填** | In/Out/非目标 |
| B2 玩家流程 | **必填**（可短） | Boot→可走→交互→存读 |
| B3 架构增量 | **必填** | 目录、程序集、Mode 占位、Boot 流 |
| B4 技术选型 | 声明沿用全局或写增量 | 禁止无 ADR 改栈 |
| B5 数据模型 | **必填最小** | schema v0、示例 def、save header |
| B6 算法/规则 | **常 N/A** | 若真无玩法公式，写 N/A 原因 |
| B7 接口契约 | **必填薄** | 用例列表即可 |
| B8 UI/UX | **必填薄** | HUD 信息架构 + 输入动作表 |
| B9 内容清单 | **必填** | placeholder、dummy 生物 |
| B10 风险 | **必填** | 3–7 条 |
| B11 验收 | **必填** | 可勾选 DoD |
| B12 任务序 | **必填** | 提交/任务顺序 |
| B13 开放问题 | 有则写 | 0–N |

## 3. VS0 Slim 官方清单（开工设计用）

VS0 设计包**必须**锁死下列战术项（来自 preflight P0）：

1. Unity **精确版本号**（如 6000.x.y LTS）与安装模块  
2. 仓库目录：`game/` `data/` `tools/` 职责  
3. 程序集边界：哪些 asmdef，Domain 无 Unity  
4. Boot → Title/AppSession → 进 R01 探索 的场景/Mode 流  
5. Input 动作表 v0（Move/Look/Interact/Pause/…）  
6. Save header + 版本字段 v0（可极少进度字段）  
7. DataCatalog 加载路径与 **1 条示例 creature def**  
8. Localization key 命名规范 v0  
9. Placeholder 资源路径约定（见 art-audio-placeholder）  
10. Git/meta/LFS 策略引用 `git-and-repo-hygiene.md`  
11. 验收手测脚本（逐步点击清单）  

**明确 N/A：** 结契判定、战斗结算、进化 DSL、经济。

## 4. 时间盒（建议）

| 活动 | 建议上限 |
|---|---|
| Slim 设计包起草 | 1 个工作日内出可读草案 |
| Full 设计包起草 | 1–2 个工作日 |
| 用户审阅 | 48h 内批 / 改 / 要补充（可协商） |
| 无阻塞意见却无限复评 | 视为流程故障，收敛到开放问题列表 |

时间盒服务节奏，不降低否决权：用户仍可随时打回。
