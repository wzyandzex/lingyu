# 世界超一流定位 · 引擎与技术栈深度复评（v1.0）

> 视角：首席技术制作人 / 游戏架构师 / 发行向工程负责人  
> 输入：产品宪法已锁定（单机根基、PC 优先、回合策略、共鸣结契、活生态、超大图鉴）  
> 目的：纠正「只为切片快」的选型偏差，回答：**若要做成能与世界顶级精灵宠物产品并列的可发行作品，技术底座应是什么。**

---

## 0. 先校准「世界超一流」在本项目里的真实含义

### 0.1 它不是什么
- 不是 UE5 宣传片级全开 Nanite/Lumen 就算一流  
- 不是系统数量最多、开放世界最大  
- 不是用最潮的技术栈证明团队很强  

### 0.2 它是什么（对本品类）
世界超一流精灵宠物产品，玩家实际记住的是：

1. **生命感**：精灵像活物，有习性、情绪、伴随表现  
2. **世界可信**：生态、天气、文化、区域气质可感  
3. **关系可感**：结契/羁绊改变体验，不是数值皮肤  
4. **手感与演出**：镜头、音频、进化、遭遇、UI 都有仪式与愉悦  
5. **内容工业力**：数百物种 + 技能 + 遭遇 + 文案仍不崩、不脏、可更新  
6. **平台完成度**：PC 设置/性能/手柄/无障碍/存档/本地化/认证级稳定  
7. **长线可运营**：资料片、世界状态、异步社交、平衡可热修  

结论：**一流首先是内容工业 + 表现纪律 + 系统深度的合成，引擎只是放大器。**  
但引擎选错，会让放大器变成减速器或风险源。

### 0.3 本品类技术真相
宝可梦/类宠物产品的胜负手通常不在「最强渲染」，而在：

- 生物动画与个性表现管线  
- 遭遇/生态/图鉴数据管线  
- 战斗可读性与平衡迭代速度  
- UI/UX 信息架构（图鉴、队伍、培养）  
- 演出编排（进化、传说、结契）  
- 大规模本地化与配置校验  

因此：  
**选引擎 = 选「内容工业化能力 + 表现工具链 + 人才/外包可获得性 + 发行路径」的总包。**

## 1. 复评方法：两层决策，而不是一个分数

### 层 A — 产品灵魂验证（Vertical Slice）
问题：能否最快、最干净地证明「共鸣世界」值得做大？

### 层 B — 世界级可发行产品（Ship & Scale）
问题：在 2–5 年生命周期里，能否支撑：
- 300–600+ 精灵与持续 DLC  
- 高质量 3D 角色/生物表现  
- PC→主机移植  
- 专业音频/过场/UI 团队协作  
- 招聘、外包、中间件、发行技术审查  

上一轮 ADR-006 偏层 A。本轮以层 B 为否决权，层 A 为约束条件。

## 2. 世界级技术需求分解（必须被栈满足）

| 域 | 世界级要求 | 技术含义 |
|---|---|---|
| 生物表现 | 剪影、骨骼、表情、伴随 AI、进化变形 | 动画状态机、Rig 规范、VFX 插槽、Timeline/Sequencer |
| 生态世界 | 天气/昼夜/季节改遭遇与行为 | 世界状态服务、数据驱动刷新、流式加载预留 |
| 结契 | 多种仪式模板、高反馈 | 状态机框架 + 可编排演出 + 输入提示系统 |
| 战斗 | 回合策略深度、可读、可观战/回放 | 纯逻辑层 + 表现适配层 + 回放序列 |
| 图鉴 | 四志分层、海量文案与解锁 | CMS 化数据、本地化管道、UI 虚拟列表 |
| 内容生产 | 非程序也能加物种 | 编辑器工具、Schema、校验 CI、预览场景 |
| 品质 | 60fps 目标、低崩溃、可追踪 | Profiling、崩溃报告、存档迁移、自动化测试 |
| 发行 | Steam/主机认证、存档、成就、平台服务 | 成熟平台插件与案例 |
| 社交（后置） | 异步展示/回放/幽灵 | 后端 API、重放格式、UGC 审核预留 |
| 协作 | 多角色并行、Agent 可维护 | 清晰模块边界、代码规范、少专有黑盒 |

## 3. 三引擎在「世界级可发行」坐标系中的真实位置

### 3.1 Godot 4.x

**世界级上的强项**
- 架构清晰，利于长期不腐化  
- 数据驱动与定制编辑器成本低  
- 授权极优（MIT），IP 与商业简单  
- 迭代快，适合打磨玩法手感  
- 对 Agent/精干团队非常友好  

**世界级上的硬伤（必须直视）**
1. **3D 生物产品的工业案例密度不足**（同品类 3D 商业标杆远少于 Unity）  
2. **中间件与外包市场小**：动画工具链、过场、平台 SDK、中间件采购选择少  
3. **主机发行路径存在但不够「默认」**：认证、TRC/Lotcheck、厂商支持成熟度弱于两家大厂  
4. **高端表现要靠自建纪律**：不是做不到一流画面，而是**同样品质下自研/自管成本更高**  
5. **招聘与团队扩展**：冲击「大制作编制」时人才池与培训成本不利  
6. **影视级编排、复杂过场、大规模动画管线**仍显著弱于 Unity/UE 生态默认能力  

**判决（层 B）**  
Godot 可以把澄界做成**极高完成度的一流独立/精品商业作**，甚至在玩法与结构上更干净。  
但若目标明确是「世界超一流、可扩主机、可扩编制、可接专业外包的主流商业大作底座」，Godot 是**高风险乐观解**，不是行业默认安全解。

### 3.2 Unity 6 / LTS

**世界级上的强项**
1. **本类产品（角色/生物/系统RPG/收集养成）事实标准底座之一**  
2. **内容工业最成熟**：ScriptableObject/Addressables、动画、Timeline、UI、输入、本地化生态  
3. **人才与外包可获得性最高**：美术/TA/程序/外包管线都好招好接  
4. **PC→主机路径最稳**  
5. **平台服务、成就、存档、SDK、分析、反作弊周边**选择最多  
6. C# 利于**领域逻辑纯净 + 单测 + 长期重构**（非常适合战斗/结契/图鉴）  
7. 足以支撑「第三人称冒险 + 强角色表现 + 大量 UI 系统」而不必 UE 的重量  

**世界级上的代价**
1. 工程极易腐化成 MonoBehaviour 泥球（**必须用架构纪律压住**）  
2. 许可与定价政策是长期治理项（需商务跟踪，不靠赌）  
3. 项目可变重，切片期若无模板纪律会慢于 Godot  
4. 包体、导入、CI 需要专业工程化  

**判决（层 B）**  
Unity 是澄界作为**世界超一流可发行精灵宠物产品**的**最优总包解**：  
表现够、内容工业强、编制可扩、主机可走、品类匹配度最高。

### 3.3 Unreal Engine 5

**世界级上的强项**
- 视听上限最高（环境、光影、Sequencer、大型场景）  
- 主机与高规格 PC 表现力顶尖  
- 适合「世界本身就是卖点」的视听导向大作  

**世界级上的错配**
1. 澄界核心差异是**共鸣/生态/关系/图鉴工业**，不是射击/动作开世界视听  
2. 回合制 + 海量配置 + 重 UI 图鉴，UE 不自动提供优势，反而加重  
3. 迭代与内容工具成本高，切片验证灵魂更慢  
4. C++/蓝图双轨在多系统 RPG 中更吃编制与规范  
5. 版税与项目重量对「先做对再做大」不友好  

**判决（层 B）**  
UE5 是「视听史诗优先」的一流解，不是「精灵宠物系统与内容工业优先」的最优解。  
仅当未来明确转型为强主机、强电影、强大世界观光主策略时再评估迁移/分仓。

## 4. 加权重评（世界级可发行权重）

| 维度 | 权重 | Godot | Unity | UE5 |
|---|---|---|---|---|
| 内容工业化（图鉴/配置/工具） | 18% | A | **S** | A |
| 生物/角色表现与动画管线 | 14% | B+ | **A/S** | S |
| 演出编排（进化/结契/过场） | 10% | B+ | **A** | **S** |
| 系统架构与可测试性 | 10% | **A/S** | A（需纪律） | B+ |
| 团队扩展/外包/人才 | 12% | B- | **S** | A |
| PC 品质与迭代 | 8% | **S** | A | B |
| 主机发行成熟度 | 8% | B | **S** | **S** |
| 长线运营与平台生态 | 6% | B | **A/S** | A |
| 授权/商务可控 | 6% | **S** | B | A |
| 与本品玩法匹配（回合+养成+探索） | 8% | A | **S** | B+ |

**综合（层 B 世界级可发行）**  
1. **Unity** — 主推荐  
2. Godot — 精品路线/精干队伍次优  
3. UE5 — 视听路线备选，非默认  

**综合（层 A 灵魂切片速度）**  
1. Godot  
2. Unity  
3. UE5  

**最终以层 B 否决权为准：主推荐从 Godot 修正为 Unity。**

## 5. 架构师结论（可执行）

### 5.1 主推荐技术底座

**Unity 6（或当前 LTS）+ URP + C# 领域层 + 数据驱动内容管线**

| 层 | 选型 | 理由 |
|---|---|---|
| 引擎 | Unity 6 / LTS | 品类匹配、工业化、主机路径 |
| 渲染 | **URP** | PC 品质与性能平衡；HDRP 对首发过重 |
| 语言 | C# | 战斗/结契/图鉴可单测、可重构 |
| 动画 | Animancer 或原生 Animator + 严格 Rig 规范 | 生物个性与伴随表现 |
| 过场/演出 | Timeline + Cinemachine | 结契/进化/传说仪式 |
| 输入 | Input System | PC 键鼠+手柄一流支持 |
| UI | **UI Toolkit**（推荐）或 UGUI 锁死一种 | 图鉴/复杂菜单长期更稳（Toolkit） |
| 内容数据 | JSON/YAML 源数据 → 导入为 SO/二进制 | 可 diff、可 CI 校验、可热更策略 |
| 资源 | 先直引用；VS 后半再 Addressables | 避免过早复杂 |
| 音频 | 原生 Audio + 预留 FMOD/Wwise 接口 | 切片先原生，音频工业升级可插 |
| 本地化 | Unity Localization 或自研表 | 图志海量文案 |
| 存档 | 原子写盘 + 版本迁移 | 长线必做 |
| 测试 | EditMode/PlayMode + 数据校验 CI | 世界级门槛 |
| 平台 | Steamworks.net 后置；主机 SDK 预留 | PC 首发 |
| 异步社交后端 | 后置：HTTPS API + 重放文件 | 不进 VS |

### 5.2 领域架构（与引擎无关的「真正一流」部分）

无论 Unity/Godot，世界级项目必须有：

```text
Presentation (View/Anim/UI/VFX)
    ↓ intents / events
Application Services (use cases)
    ↓
Domain (Battle/Bonding/Codex/Affinity/Evolution)  ← 纯逻辑，可单测
    ↓
Data (defs) + Save (state) + WorldClock
```

**硬规则**
- 战斗结算、结契判定、图鉴解锁、进化条件：**零** Unity API 依赖  
- MonoBehaviour 只做生命周期与表现绑定  
- 新增一只精灵：加数据 + 资源 + 本地化，不改核心代码  
- 回放/观战/异步幽灵：建立在「指令序列」而非随机帧同步  

这套架构，才是世界一流，而不是引擎 Logo。

### 5.3 内容工业栈（比引擎更决定上限）

| 管线 | 世界级做法 |
|---|---|
| 精灵 | DCC（Blender/Maya）→ 规范 Rig → 导入校验 → 预览场景 |
| 技能 | 数据定义 + 时间轴事件（hit/fx/cam） |
| 遭遇 | 区域表 × 天气 × 时段 × 季节 |
| 文案 | 表驱动 + 双文体（科普/诗意）+ 审核状态 |
| 平衡 | 可脚本导出计算、批量模拟（战斗蒙特卡洛后置） |
| CI | schema 校验、失踪引用、本地化缺 key、命名规范 |

没有这条工业线，Unity 也会做出平庸大垃圾；有这条线，Godot 也能出精品。  
**但 Unity 让这条线「雇得到人、买得到工具、接得住外包」。**

## 6. 对上一轮 Godot 推荐的修正说明

上一轮推荐 Godot 的前提是：  
「当前最大风险是独特性未验证，应最大化切片速度。」

这个前提**仍然正确**，但在用户明确要求以**世界超一流游戏架构师**重评后，必须补上第二前提：

> 技术选型不能只服务前 3 个月，必须服务未来 36 个月的可发行与可扩展。

因此：

| 阶段 | 策略 |
|---|---|
| 文档与规格 | 已完成，与引擎无关 |
| 实现起点 | **直接 Unity**，避免双引擎浪费 |
| 切片速度 | 用「强力项目模板 + 纯领域层 + 少插件」追回 Godot 的敏捷 |
| 不建议 | Godot 做完切片再整体搬 Unity（迁移税极高，历史项目常见失败） |

**唯一仍应选 Godot 的情况**
- 团队永久精干（核心 < 5–8 人）且明确走精品独立发行  
- 强烈拒绝 Unity 商务条款  
- 主机不是中期目标  
- 接受自建更多工具与表现管线  

若你要的是「世界超一流、可扩编制、可接主流商业制作」——**选 Unity。**

## 7. 完整技术栈推荐清单（v1 锁定草案）

### 7.1 运行时
- Unity 6 / LTS  
- URP  
- C# 9/10+（随编辑器）  
- Input System  
- Cinemachine  
- Timeline  
- AI Navigation（探索 NPC/轻量）按需  
- UniTask（推荐，异步卫生）  
- 可选：MessagePipe / 自研轻事件总线  

### 7.2 内容与数据
- `data/` 源文件 JSON（Git 友好）  
- 编辑器导入器生成 `ScriptableObject` 或 `Binary`  
- JSON Schema + Python/Node 校验  
- 本地化 CSV/表 → 运行时 string table  

### 7.3 工程与品质
- 程序集拆分：`Domain` `Application` `Unity.Presentation` `Unity.Editor`  
- EditMode tests（Domain 全覆盖关键规则）  
- 命名/目录规范（见 engineering skeleton Unity 版）  
- 崩溃与日志：Unity Cloud Diagnostics 或等价（后置）  
- Addressables：VS6 后引入  

### 7.4 音频
- Phase VS：Unity Audio Mixer  
- Phase 产品化：评估 **FMOD**（更常见于中大型）  

### 7.5 后端（仅异步社交阶段）
- 最小：对象存储 + 鉴权 API + 回放元数据  
- 不做实时 MMO 帧同步  

### 7.6 DCC
- 建模/绑定：Blender 为主（成本与脚本友好）或 Maya（外包主流）  
- 贴图：Substance  
- 音频 DAW：Audacity/Reaper 等（制作侧）

## 8. 风险与缓解（Unity 主路径）

| 风险 | 缓解 |
|---|---|
| 架构腐化 | 强制 Domain 程序集无 Unity 引用；Code Review 否决上帝对象 |
| 切片变慢 | 官方/自研 Slim Template；禁止 Addressables/HDRP/重插件开局 |
| 许可政策 | 商务季度 review；收入模型与 Unity 条款对齐检查 |
| 过度资产店依赖 | 核心系统自研；资产只用于 placeholder 与非核心 |
| UI 选型摇摆 | 开工前锁 UI Toolkit 或 UGUI，禁止双轨 |
| 动画品质不够 | 第一天定 Rig/导出规范；旗舰精灵预算倾斜 |
