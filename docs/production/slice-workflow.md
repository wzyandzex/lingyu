# 切片工作协议（Slice Workflow Protocol）v1.0

> **状态**：强制  
> **目的**：每一个 VS 里程碑都按世界一流标准「先设计后实现、先验收再前进」  
> **服从**：`AGENTS.md` §10B、`PRD-vertical-slice.md`、ADR-001～008  
> **原则**：用户可在任意阶段提出意见；未获实现许可前禁止写业务实现代码

---

## 0. 你与 Agent 如何协作（人话版）

```text
Agent 宣告：现在是哪个切片、要做什么、是否需要新设计
    → Agent 产出设计包（架构/选型/数据/算法/验收）并对比方案
    → 你提意见 / 改方向 / 追问
    → Agent 修订设计
    → 你明确允许实现（或说「按方案开工」）
    → Agent 实现
    → Agent 自测 + 对照验收清单报告
    → 你验收通过 / 打回
    → 只有通过才进入下一切片
```

**是的**：每个切片实现前，凡涉及新边界/新模型/新算法/新技术点，都应先对比评估、锁定设计，再动手。  
**不是**：每个切片都从零重选引擎或推翻总体架构——全局栈与总架构已锁，切片只做**增量设计**。

## 2. 每个切片的强制阶段

### Phase A — 宣告（Announce）
Agent **必须先说清**，不得默认开码：

1. **当前切片 ID**（如 VS2）  
2. **切片目标**（玩家体验一句话）  
3. **映射的 PRD 需求 ID / 用户故事**  
4. **依赖**：哪些前序切片必须已通过  
5. **本切片是否引入新设计点**（架构增量 / 选型 / 数据模型 / 算法）  
6. **建议状态**：仅设计 / 可设计并待你批准实现 / 阻塞中  

### Phase B — 设计（Design Gate）— 实现前必过

产出文件（建议路径）：

```text
docs/production/slices/VS{N}-design.md
```

设计包**最低目录**（按世界一流标准，可裁剪但需声明「不适用」原因）：

| 章节 | 必须回答 |
|---|---|
| B1 范围 | In/Out；非目标；防膨胀清单 |
| B2 玩家流程 | 主路径步骤；失败/重试；时长预算 |
| B3 架构增量 | 新/改模块、Mode、上下文、事件、与 §7A 对齐 |
| B4 技术选型 | 仅本切片增量；对比 ≥2 方案时写推荐+否决理由；禁止无 ADR 改全局栈 |
| B5 数据模型 | def/state 字段、ID、存档影响、schema 变更 |
| B6 算法/规则 | 状态机、公式、伪代码、复杂度、可测点 |
| B7 接口契约 | Application 用例、Domain API、事件列表 |
| B8 UI/UX | 信息架构、反馈链、教学负担 |
| B9 内容清单 | 精灵/关卡/文案/音频占位 |
| B10 风险 | 技术/体验/范围风险与缓解 |
| B11 验收 | 本切片 DoD（可勾选） |
| B12 实现任务序 | 建议 commit/任务切片顺序 |
| B13 开放问题 | 需你拍板的 0–N 项 |

**对比评估规则**
- 有关键分叉时：给 2–3 案，附「世界一流选择」推荐与代价  
- 无分叉时：写「沿用总架构/总栈，无新选型」并说明为何足够  
- 算法/状态机必须可单测或说明为何仅手测  

**设计门禁（未过禁止实现）**
- [ ] 映射 PRD，无孤儿功能  
- [ ] 不破坏 Domain 纯净与数据驱动  
- [ ] 数据/存档影响写清  
- [ ] 验收可观察  
- [ ] 用户未提出未解决的阻塞意见  

### Phase C — 用户确认（User Gate）

Agent 必须明确提问或等待等价许可，例如：

- 「VS{N} 设计包已就绪，是否批准实现？」  
- 或你主动说：「按方案开工 / 批准 VS{N}」  

**许可前禁止**
- 业务功能实现（可做文档、目录空壳说明、非侵入性工具草稿需声明）  
- 以“先写着以后改”绕过设计门  

**你可随时**
- 改需求、改优先级、否决方案、要求补对比、要求降范围保品质  

Agent 收到意见后：先更新设计包/PRD，再谈实现。

### Phase D — 实现（Implement）

1. 严格按批准的设计包与 VS PRD  
2. 小步提交逻辑；关键规则先测后堆表现（适合的领域）  
3. 偏离设计的必要改动：**先记录再改**，重大偏离停下来同步你  
4. 不顺手做下一切片功能  

### Phase E — 实现后验收（Review Gate）

Agent 必须交付：

| 交付 | 内容 |
|---|---|
| 变更摘要 | 做了什么 / 没做什么 |
| 对照验收清单 | B11 / PRD 本切片条目逐项 ✅/❌ |
| 测试报告 | 单测/手测结果；失败如实 |
| 已知问题 | 分级：阻断/严重/轻微 |
| 架构合规自检 | §7A 否决项是否触碰 |
| 设计偏差表 | 与设计包差异及原因 |
| 下一切片建议 | 是否建议进入 VS{N+1} 的宣告 |

**通过标准**
- 本切片 DoD 全过（或你书面豁免某项并记录风险）  
- 无阻断 bug  
- 未批准的范围膨胀 = 不通过  

**未通过**
- 不得开始下一切片实现  
- 回到设计修订或实现返工  

### Phase F — 收尾与基线（Baseline）

通过后：
1. 更新 `docs/production/slices/STATUS.md`  
2. 必要时回写 PRD/架构「已落地」注释  
3. 标记 VS{N} = Accepted Baseline  
4. 才允许 Phase A of VS{N+1}

## 3. 什么叫「本切片需要重新做技术选型/架构」

| 情况 | 动作 |
|---|---|
| 沿用 Unity/URP/UI Toolkit/总分层 | 声明无新全局选型 |
| 新 Mode 内模块拆分、新状态机 | **要做**切片架构增量设计 |
| 新算法（结契判定、伤害、遭遇权重） | **要做**算法设计+可测点 |
| 新第三方包 | 包门禁 + 设计包对比 + 必要时 ADR |
| 改全局栈（如换 UI 方案） | **完整 ADR**，升级到项目级，不单藏在切片里 |

## 4. Agent 每次开口的推荐模板

### 4.1 切片宣告模板

```markdown
## 当前切片：VS{N} — {名称}

**目标**：{玩家体验一句话}
**PRD 映射**：{需求 ID 列表}
**依赖**：VS{N-1} 已通过？{是/否}
**本切片设计点**：
- 架构增量：{有/无 + 要点}
- 技术选型：{有/无 + 要点}
- 数据模型：{有/无 + 要点}
- 算法/规则：{有/无 + 要点}

**当前阶段**：宣告 / 设计中 / 待你确认 / 实现中 / 待验收
**建议**：先完成设计包，请你审阅后再实现。
```

### 4.2 请求实现许可模板

```markdown
## VS{N} 设计门完成 — 请求实现许可

设计包：docs/production/slices/VS{N}-design.md
推荐方案：{一句话}
关键分叉与你的可选意见：{0–N}
验收摘要：{3–6 条}

请回复：
- 「批准 VS{N} 实现」或
- 修改意见
```

### 4.3 实现后报告模板

```markdown
## VS{N} 实现后报告

### 验收对照
| ID | 结果 | 说明 |
|---|---|---|

### 测试
### 已知问题
### 设计偏差
### 是否建议进入 VS{N+1}
等待你验收通过。
```

## 5. 与总体垂直切片（60–90 分钟）的关系

- **微切片** VS0–VS6：工程与系统增量单位（本协议主对象）  
- **体验切片** 全文：VS6 通过后的整包体验验收（E1–E7 等）  

禁止：VS2 未过就宣称“垂直切片做完了”。  
禁止：跳过 VS 设计门直接打体验包装。
